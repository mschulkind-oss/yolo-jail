## 1. [HIGH] split_env keeps the entire nix stdenv toolchain, not just declared packages, shadowing the macOS system userland
- **where:** `src/cli/darwin_packages.py:142`  (correctness)
- **detail:** split_env partitions PATH by keeping every entry that startswith('/nix/store/'). But the PATH it is fed comes from `nix print-dev-env .#devShells.aarch64-darwin.yoloDarwinPackages`, and the devShell is a plain `pkgs.mkShell { packages = darwinPackages; }` (flake.nix:672-674). `print-dev-env` reproduces the full `nix develop` build environment, so its PATH contains the entire aarch64-darwin stdenv toolchain — the clang cc-wrapper, cctools `ld`, GNU coreutils, gnused, gnugrep, gawk, gnumake, findutils, tar/gzip/xz, patch, diffutils, etc. — ALL under /nix/store, IN ADDITION to the declared package bin dirs. split_env therefore keeps all of them. The module comment (lines 138-140) and docstring claim it contributes 'only the package store dirs', and the unit tests (tests/test_darwin_packages.py:87, :135) encode that wrong mental model with only 1-2 clean package bin dirs, so they pass while the real behavior is a full-toolchain PATH injection. Because sandbox_path() inserts this prefix BEFORE /usr/bin AND /bin (macos_user.py:488-501), the nix GNU/clang toolchain shadows the macOS system userland for the agent. Compounding it: ENV_WHITELIST (line 53) deliberately strips NIX_CFLAGS_COMPILE / NIX_LDFLAGS / NIX_CC etc., so the nix cc-wrapper that now shadows /usr/bin/cc is left without the env it depends on. A correct implementation must subtract the empty-devShell (stdenv) PATH baseline — as `nix develop` itself does when computing added entries — or read the declared buildInputs' bin dirs directly, rather than keep every /nix/store PATH entry.
- **failure:** User sets packages: ["jq"] with runtime: macos-user on a Mac. materialize() runs print-dev-env on the darwin mkShell; the returned PATH contains jq's bin dir PLUS the whole aarch64-darwin stdenv (clang wrapper, cctools ld, GNU coreutils, gnused, gnugrep, gawk, gnumake, ...). split_env keeps all of them and they are prepended ahead of /usr/bin and /bin in the sandbox agent's PATH. Inside the sandbox the agent now runs nix GNU `sed`/`grep`/`awk`/`ls` instead of macOS BSD versions (behavioral divergence for any script assuming BSD userland), and a nix `cc`/`clang` shadows /usr/bin/cc while the NIX_* env it needs was stripped by the whitelist, so compiles invoked through it can fail — directly contradicting the module's promise to contribute only the declared package store dirs.
- **evidence:** I reproduced the behavior end-to-end with real nix (2.34.7) in the jail. The flake's darwin devShell is a plain `pkgs.mkShell { packages = darwinPackages; }` (flake.nix:672-674). I built an equivalent `mkShell { packages = [ pkgs.jq ]; }`, ran the CLI's own `print_dev_env_argv` invocation (`nix print-dev-env --json`), and fed the parsed output through the actual `cli.darwin_packages.split_env`:

- PATH from print-dev-env had 22 entries; `split_env` kept ALL 22 (every one is under /nix/store). Only ONE (jq-1.8.2-bin/bin) is the declared package. The other 21 are the full stdenv build toolchain: gcc-wrapper, gcc, glibc-bin, coreutils, binutils-wrapper, binutils, findutils, diffutils, gnused, gnugrep, gawk, gnutar, gzip, bzip2, gnumake, bash, patch, xz, file, patchelf. On aarch64-darwin these are the clang/cctools equivalents, but the structural fact is identical — mkShell's `print-dev-env` reproduces the whole build environment, and split_env's filter `p.startswith('/nix/store/')` (darwin_packages.py:142-146) keeps every one.

- These 21 toolchain dirs ride unfiltered through `darwin_prefix` → `launch_argv(path_prefix=darwin_prefix)` (macos_user.py:863, 908) into `sandbox_path`, which inserts `*(prefix or [])` at line 495 — AFTER the agent shim dirs but BEFORE `/usr/bin` and `/bin` (lines 496-497). So nix gnused/gnugrep/gawk/coreutils/cc shadow the macOS BSD userland for the sandboxed agent. No downstream filtering exists between materialize and PATH.

- The compounding NIX_* claim also holds: the dev-env contained NIX_CC, NIX_CFLAGS_COMPILE, NIX_LDFLAGS, NIX_BINTOOLS, NIX_HARDENING_ENABLE, etc., but `split_env` returned `extra == {}` because ENV_WHITELIST is only `('PKG_CONFIG_PATH',)` (line 53). So the cc/gcc-wrapper now on PATH is stripped of the env it needs.

The module docstring (lines 138-140: "we contribute only the package store dirs") and tests (test_darwin_packages.py:87 asserts prefix == exactly the 1-2 clean bin dirs, :100 asserts only PATH is store dirs) encode the wrong mental model with hand-picked 1-2 entry PATHs, so they pass while real behavior injects the full toolchain. The handoff doc itself flags "print-dev-env dumps the full stdenv env" (line 376) yet the code never subtracts an empty-shell baseline.
- **fix:** Subtract the stdenv baseline the way `nix develop` does: materialize an empty `mkShell {}` (no packages) devShell's print-dev-env PATH and remove those store dirs from the yoloDarwinPackages PATH, keeping only the delta (the declared packages' bin dirs). Alternatively, read the declared buildInputs' `bin` dirs directly (e.g. `nix build`/`nix eval` the package outPaths and append `/bin`) instead of scraping the dev-env PATH. Either approach yields only the declared `packages:` on the sandbox PATH, preserving the macOS BSD userland ahead of nix GNU tools.

## 2. [HIGH] Unresolvable pinned nixpkgs rev aborts the entire darwin eval — warn-and-skip defeated, valid sibling packages lost
- **where:** `flake.nix:184`  (flake.nix darwin block — eval correctness of warn-and-skip, spec branches, and cross-path/Linux-eval safety)
- **detail:** In darwinResolved, for a pinned spec the `src` attrset is built by `import (builtins.fetchTarball { url = ".../archive/${spec.nixpkgs}.tar.gz"; }) { inherit system; }` (lines 183-188). This fetchTarball is forced immediately by the very next guard `present = src ? ${attr}` (line 193). A fetchTarball failure (typo'd rev, deleted commit, offline host, GitHub 404/5xx) throws a builtin evaluation error that (a) is NOT inside any tryEval, and (b) CANNOT be caught by tryEval even if it were — tryEval only catches `throw`/`assert`, not builtin fetch/IO errors. I verified this directly: `builtins.tryEval (import (fetchTarball <bad-rev>) {})` still aborts nix with exit 1. The code comment on lines 190-192 asserts the `?` guard makes unknown packages skippable; that is true only for attribute-missing, and it papers over the fact that the fetch on the preceding lines is the actual failure mode. Because a single bad pin aborts eval of BOTH `devShells.yoloDarwinPackages` and `darwinUnavailablePackages`, materialize()/_skipped_names both fail, run_macos_user raises DarwinPackagesError, and the whole run aborts — the exact opposite of the advertised warn-and-skip. The image path (line 142-144) shares this pattern, but it never promised warn-and-skip; the darwin block does.
- **failure:** User sets packages: ["ripgrep", {"name":"jq","nixpkgs":"deadbeef..."}] (mistyped/deleted rev, or CI runs offline). VERIFIED with nix: `YOLO_EXTRA_PACKAGES='["ripgrep",{"name":"jq","nixpkgs":"0000..."}]' nix eval .#darwinUnavailablePackages.aarch64-darwin` exits 1 with an uncatchable 404 error; `nix build .#devShells.aarch64-darwin.yoloDarwinPackages` likewise fails. The valid ripgrep is never materialized and the macos-user run aborts entirely, instead of skipping jq with a warning.
- **evidence:** All three load-bearing claims verified against the actual code and empirically with nix 2.34.7.

1. tryEval cannot catch a fetchTarball IO/404 error. Direct test in /tmp:
   `builtins.tryEval (let x = import (builtins.fetchTarball {url=".../0000...tar.gz"}) {}; in x ? hello)`
   → nix exits 1 with "HTTP error 404" escaping tryEval entirely (not converted to {success=false}).

2. In flake.nix:183-193 the pinned-spec `src` = `import (builtins.fetchTarball {url=".../${spec.nixpkgs}.tar.gz"}) {inherit system;}` is forced to WHNF by the very next line `present = src ? ${attr}` (line 193). This computation sits OUTSIDE the only tryEval in the block (line 210-213, which wraps just `pkgs.lib.meta.availableOn`). So the fetch failure is neither wrapped nor catchable.

3. End-to-end flake eval with a real triggering input:
   `YOLO_EXTRA_PACKAGES='["ripgrep",{"name":"jq","nixpkgs":"0000000000000000000000000000000000000000"}]' nix eval --impure .#darwinUnavailablePackages.aarch64-darwin`
   → exits 1. Stack trace points at flake.nix:675 → 222 (`darwinSkippedNames = map ... darwinResolved`) → the 404. Because `builtins.filter`/`map` force every element of `darwinResolved`, the single bad pin aborts evaluation of the whole list; the valid `ripgrep` sibling is never materialized. `.#devShells.aarch64-darwin.yoloDarwinPackages` (darwinPackages, line 223-224) shares the same forced list and fails identically.

Run-abort path confirmed: darwin_packages.materialize (src/cli/darwin_packages.py:190-206) runs `print-dev-env`; returncode!=0 → raise DarwinPackagesError. macos_user.py:1149-1157 catches it and `return 1` — the run aborts entirely instead of warn-and-skipping. (_skipped_names at 157-176 is best-effort and swallows its own failure, but print-dev-env is the fatal call.)

The offline variant is even stronger than the typo case: an offline/CI host fails the fetch even for a CORRECT pin, taking down all locally-cached sibling packages too.

One fairness caveat on framing: the code comment (flake.nix:169-173) advertises warn-and-skip specifically for "a package with no darwin build" (platform-unavailability via availableOn), not explicitly for unresolvable pins, and the image path (line 142-144) has the identical fetch-abort behavior as an established contract. So "the exact opposite of advertised warn-and-skip" is a slight overstatement of scope. But the finding's core defect is real and verified: the comment at lines 190-192 overstates the `?` guard's protection ("what makes an unknown package skippable"), and a single unresolvable/unreachable pin forces an uncatchable eval abort that discards all valid sibling packages and kills the run.
- **fix:** Force the fetch inside a catchable boundary. builtins.tryEval cannot catch fetchTarball errors, so tryEval is not the tool. Options: (a) resolve pinned nixpkgs as flake inputs / via flake.lock so an unresolvable rev is a lock-time error, not an eval-time abort of unrelated packages; (b) move pinned-spec resolution out of the darwinResolved list so a fetch failure for one pin can't force the whole map/filter (e.g. resolve each pin in isolation and represent failure as `available = false` with the name recorded in darwinSkippedNames — but note IO errors still can't be caught in pure Nix, so this really needs (a) or a host-side pre-check); (c) at minimum, correct the comment at flake.nix:190-192 to stop claiming the `?` guard makes pinned specs skippable, since the preceding fetchTarball is the uncatchable failure mode.

## 3. [HIGH] split_env leaks the entire nix stdenv toolchain onto the sandboxed agent's PATH ahead of /usr/bin
- **where:** `src/cli/darwin_packages.py:142`  (correctness — acceptance-bar trace (config packages → native darwin nix → sandbox PATH))
- **detail:** split_env keeps EVERY /nix/store/*/bin entry from the devShell's print-dev-env PATH. But devShells.yoloDarwinPackages is `pkgs.mkShell { packages = darwinPackages; }`, and mkShell unconditionally injects the full stdenv build toolchain. I confirmed this empirically: an EMPTY mkShell (`packages = []`) already puts 21 store bin dirs on PATH — coreutils, gnused, gnugrep, gawk, gnutar, gzip, gnumake, bash, patch, diffutils, findutils, gcc/binutils, xz, bzip2, file. All of these pass the `startswith('/nix/store/')` filter and become darwin_path_prefix. sandbox_path (line 495) inserts the prefix BEFORE /usr/bin and /bin, so on a real run these GNU tools shadow the macOS BSD system tools for the sandboxed agent. This directly contradicts the module's own comment on line 139-140 ('we contribute only the package store dirs') and is inconsistent with the strict ENV_WHITELIST the code carefully applies to non-PATH vars (lines 43-53) precisely to avoid stdenv pollution — the PATH channel gets no such filtering. The acceptance bar for package *presence* is met, but the agent silently inherits an entire undeclared toolchain.
- **failure:** Config declares packages: ["jq"]. On a real macOS run, the agent's PATH gets jq's bin plus ~21 stdenv dirs ahead of /usr/bin. The agent (or a user script) runs `sed -i '' -e 's/a/b/' f` expecting BSD sed; it instead resolves to GNU sed, which treats '' as the script and behaves differently/errors. Same class of breakage for grep, awk, tar, make, and `sh`/`bash` all silently resolving to nix stdenv builds instead of the system ones.
- **evidence:** The mechanism the finding rests on is real and I reproduced it. devShells.yoloDarwinPackages is `pkgs.mkShell { packages = darwinPackages; }` (flake.nix:672-674). I built an EMPTY `pkgs.mkShell { packages = []; }` against a pinned nixpkgs and ran the exact same command materialize() uses (`nix print-dev-env --json`). The exported PATH had 21 entries, ALL of them `/nix/store/*/bin`: patchelf, gcc-wrapper, gcc, glibc-bin, coreutils, binutils(-wrapper), findutils, diffutils, gnused, gnugrep, gawk, gnutar, gzip, bzip2, gnumake, bash, patch, xz, file. split_env's only filter is `p.startswith('/nix/store/')` (darwin_packages.py:142-146), so every one of these 21 stdenv-toolchain dirs is kept as path_prefix — with ZERO declared packages. On darwin the stdenv toolchain is clang/cctools instead of gcc/binutils, but the GNU userland (coreutils/gnused/gnugrep/gawk/gnutar/gnumake/bash/findutils/diffutils) is identical, so the shadowing holds.

Full trace confirmed against actual code:
- materialize() → split_env() returns path_prefix (darwin_packages.py:211-212)
- build_run_plan: darwin_prefix = list(darwin.path_prefix) (macos_user.py:863), passed as launch_argv(path_prefix=darwin_prefix) (macos_user.py:903-908)
- launch_argv → sandbox_path(home, path_prefix) (macos_user.py:535)
- sandbox_path inserts `*(prefix or [])` at macos_user.py:495 — AFTER the agent shim dirs but BEFORE `/usr/bin`/`/bin` (lines 496-497)

So the store GNU tools sit ahead of /usr/bin on the sandboxed agent's PATH. This directly contradicts the module's own comment ('we contribute only the package store dirs', line 139) and is inconsistent with the deliberate ENV_WHITELIST applied to non-PATH vars (lines 43-53) to avoid exactly this stdenv pollution — the PATH channel gets no equivalent filter. The failure scenario is accurate: with `packages: ["jq"]`, the agent gets jq's bin PLUS ~21 stdenv dirs ahead of /usr/bin; `sed -i '' ...` resolves to GNU sed (treats '' as script) instead of BSD sed, and sh/bash/grep/awk/tar/make silently resolve to nix builds. Acceptance bar for package *presence* is met, but an entire undeclared toolchain rides along and shadows macOS system tools.
- **fix:** split_env must not keep every /nix/store bin dir. Options: (1) materialize an empty-shell baseline PATH and subtract it, keeping only the delta introduced by darwinPackages; (2) abandon mkShell/print-dev-env for a `pkgs.buildEnv`/`pkgs.symlinkJoin` of darwinPackages and read only that profile's bin dir; or (3) build a profile via `nix build .#...` and thread only the resulting out/bin. Any of these confines the prefix to the declared packages, matching the module's stated contract and the ENV_WHITELIST discipline.

Thread that JSON through build_run_plan → entrypoint_bootstrap_script the same way YOLO_AGENTS is threaded, so generate_shims() at line 647 sees it and materializes the block/filter shims. Optionally add a plan_invariants guard asserting YOLO_BLOCK_CONFIG is present in the bootstrap text, mirroring the existing git-identity guard at macos_user.py:963-964.

## 6. [MEDIUM] Native darwin package build runs before sandbox-exec / sandbox-user preconditions
- **where:** `src/cli/macos_user.py:1150`  (correctness)
- **detail:** In run_macos_user, darwin_packages.materialize() is invoked (line 1150) — which can trigger a cold from-source aarch64-darwin build with a 1800s timeout (darwin_packages.py:197) — BEFORE the preconditions that check `shutil.which('sandbox-exec')` (line 1175) and `_sandbox_user_exists()` (line 1181). On a host where `yolo macos-setup` was never run (no sandbox user), the user waits through a potentially multi-minute nix toolchain/package build only to be told the sandbox user does not exist and to run macos-setup. The cheap, always-failing preconditions should be checked before the expensive materialize step.
- **failure:** A user configures packages: [<something needing a source build>] and runtime: macos-user but has not yet run `yolo macos-setup`. `yolo` (non-dry-run) spends many minutes building the darwin closure via print-dev-env, then hits the `_sandbox_user_exists()` check, prints 'Sandbox user _yolojail does not exist', and returns 1 — the whole build was wasted on a run that could have been rejected in milliseconds.
- **evidence:** In `run_macos_user` (src/cli/macos_user.py), the control flow past the dry-run branch is exactly:
- L1131 `_is_macos()` guard (passes on any Mac).
- L1147-1150 `pkgs = _effective_packages(config)`; if non-empty, `darwin_packages.materialize(repo_src.parent, pkgs)` is called.
- L1166 `plan_invariants`.
